    Multi-Node Setup Manual

    This manual provides comprehensive instructions for setting up and running multi-node distributed training with the AI Computing Broker (ACB) and Adaptive GPU Allocator (AGA) system.

    Overview

    Multi-node distributed training allows you to scale your machine learning workloads across multiple machines, each equipped with one or more GPUs. The ACB/AGA system provides dynamic GPU allocation and resource management across these nodes.

    Components:

    • Controller Node: The primary coordination node that manages distributed training coordination
    • Executor Node(s): Additional compute nodes that participate in distributed training
    • GPU Assigner Service: Background service running on each node to manage GPU resources
    • Training Process: Your actual machine learning training script executed via torchrun

    Prerequisites

    System Requirements

    • Operating System: Linux (Ubuntu 18.04+ recommended)
    • Python: 3.9 or higher
    • PyTorch: 2.5.1 (specific version required)
    • Network: TCP connectivity between all nodes
    • Privileges: sudo access for system configuration

    Required Tools

    Install these tools on all nodes:

    # Install jq for JSON processing
sudo apt-get update
sudo apt-get install jq

# Optional: create a Python virtual environment
python -m venv .venv
source .venv/bin/activate

# Install PyTorch 2.5.1 (exact version required)
pip install torch==2.5.1

# Install ACB/AGA system (replace with actual path)
pip install -e /path/to/adaptive-gpu-allocator/

    Network Configuration

    Ensure the following ports are available and not blocked by firewalls:

    • GPU Assigner Service Port: Default 11234 (configurable)
    • PyTorch Rendezvous Port: Default 29400 (configurable)

    Test connectivity between nodes: 

    # Test basic connectivity
ping <target_node_ip>

# Test specific port connectivity
nc -zv <target_node_ip> <port_number>

    Step 1: Environment Setup

    1.1 Gather Node Information

    On each node, collect the required network information:

    # Get IP address
hostname -I

# Get hostname
hostname

    Record this information as you'll need it for configuration.

    1.2 Create Configuration File

    Create a config.json file with your specific network and training parameters (replace the IP addresses, hostnames, and paths with your actual values):

    {
  "network": {
    "controller": {
      "ip": "192.168.1.100",
      "hostname": "controller-node"
    },
    "executor": {
      "ip": "192.168.1.101", 
      "hostname": "executor-node"
    },
    "ports": {
      "gpu_assigner": 11234,
      "rdzv": 29400
    }
  },
  "training": {
    "total_epochs": 100,
    "save_every": 10,
    "rdzv_id": 456,
    "nnodes": 2,
    "nproc_per_node": 1,
    "script_path": "path/to/your/training/script.py"
  }
}

    Configuration Parameters

    Parameter Description Example Value
    network.controller.ip Controller node IP address "192.168.1.100"
    network.controller.hostname Controller node hostname "controller-node"
    network.executor.ip Executor node IP address "192.168.1.101"
    network.executor.hostname Executor node hostname "executor-node"
    network.ports.gpu_assigner GPU assigner service port 11234
    network.ports.rdzv PyTorch rendezvous port 29400
    training.total_epochs Total training epochs 100
    training.save_every Checkpoint save frequency (epochs) 10
    training.rdzv_id Unique rendezvous identifier 456
    training.nnodes Total number of nodes 2
    training.nproc_per_node Processes per node (usually 1) 1
    training.script_path Path to training script "samples/training_script.py"

    Step 2: Node Configuration

    This step configures each node's system settings and generates the required configuration files.

    2.1 Extract Configuration and Create ACB Directory

    On all nodes, extract configuration values and create the ACB configuration directory:

    # Ensure config.json exists in current directory
ls config.json

# Extract configuration values from config.json
CONTROLLER_IP=$(jq -r '.network.controller.ip' config.json)
EXECUTOR_IP=$(jq -r '.network.executor.ip' config.json)
CONTROLLER_HOSTNAME=$(jq -r '.network.controller.hostname' config.json)
EXECUTOR_HOSTNAME=$(jq -r '.network.executor.hostname' config.json)
GPU_ASSIGNER_PORT=$(jq -r '.network.ports.gpu_assigner' config.json)

# Determine ACB directory path
if [[ -n "${SUDO_USER:-}" ]]; then
    ACB_DIR="/home/$SUDO_USER/.acb"
else
    ACB_DIR="$HOME/.acb"
fi

# Create ACB configuration directory
mkdir -p "$ACB_DIR"
echo "ACB directory created at: $ACB_DIR"

# Generate multinode.toml configuration file
cat > "$ACB_DIR/multinode.toml" << EOF
[$CONTROLLER_HOSTNAME]
type = "controller"
ipaddr = "$CONTROLLER_IP"
logfile = "$ACB_DIR/gpu-assigner.$CONTROLLER_HOSTNAME.log"
pidfile = "$ACB_DIR/gpu-assigner.$CONTROLLER_HOSTNAME.pid"
port = $GPU_ASSIGNER_PORT

[$EXECUTOR_HOSTNAME]
type = "executor"
ipaddr = "$EXECUTOR_IP"
logfile = "$ACB_DIR/gpu-assigner.$EXECUTOR_HOSTNAME.log"
pidfile = "$ACB_DIR/gpu-assigner.$EXECUTOR_HOSTNAME.pid"
port = $GPU_ASSIGNER_PORT
EOF

echo "Generated: $ACB_DIR/multinode.toml"

    2.2 Update System Hosts File

    Using the variables from step 2.1:

    On the controller node, add hostname mapping:

    # Remove any existing entries for this hostname
sudo sed -i "/^[[:space:]]*[0-9.]*[[:space:]]*$CONTROLLER_HOSTNAME[[:space:]]*$/d" /etc/hosts

# Add new hostname mapping
echo "$CONTROLLER_IP $CONTROLLER_HOSTNAME" | sudo tee -a /etc/hosts

# Verify the entry was added
grep "$CONTROLLER_HOSTNAME" /etc/hosts

    On the executor node, add hostname mapping:

    # Remove any existing entries for this hostname
sudo sed -i "/^[[:space:]]*[0-9.]*[[:space:]]*$EXECUTOR_HOSTNAME[[:space:]]*$/d" /etc/hosts

# Add new hostname mapping
echo "$EXECUTOR_IP $EXECUTOR_HOSTNAME" | sudo tee -a /etc/hosts

# Verify the entry was added
grep "$EXECUTOR_HOSTNAME" /etc/hosts

    2.3 Validate Configuration

    Verify the configuration files are correct:

    # Check if config.json is valid JSON
jq -e '.' config.json >/dev/null && echo "config.json is valid" || echo "config.json is invalid"

# Check if multinode.toml was created
ls -la ~/.acb/multinode.toml

# Display the generated configuration
cat ~/.acb/multinode.toml

    Step 3: Start GPU Assigner Services

    IMPORTANT: The GPU Assigner services must be started in a specific order: Executor nodes first, then Controller node.

    3.1 Start Executor Node Service

    On the executor node, start the GPU Assigner service:

    # Extract port from configuration
GPU_ASSIGNER_PORT=$(jq -r '.network.ports.gpu_assigner' config.json)

# Start GPU Assigner service on executor node
gpu_assigner --address 0.0.0.0 --port $GPU_ASSIGNER_PORT --multinode start --scheduler gpu-affinity

# Verify the service started successfully
gpu_assigner status

# Check if the process is running
ps aux | grep gpu_assigner

# Verify port is listening
ss -tlnp | grep $GPU_ASSIGNER_PORT

    Expected output should show the GPU Assigner service running and listening on the specified port.

    3.2 Start Controller Node Service

    On the controller node (only after executor service is running), start the GPU Assigner service:

    # Extract port from configuration
GPU_ASSIGNER_PORT=$(jq -r '.network.ports.gpu_assigner' config.json)

# Start GPU Assigner service on controller node
gpu_assigner --address 0.0.0.0 --port $GPU_ASSIGNER_PORT --multinode start --scheduler gpu-affinity

# Verify the service started successfully
gpu_assigner status

# Check if the process is running
ps aux | grep gpu_assigner

# Verify port is listening
ss -tlnp | grep $GPU_ASSIGNER_PORT

    3.3 Verify Multi-Node Communication

    Test that the nodes can communicate with each other:

    # From controller node, test connection to executor
EXECUTOR_IP=$(jq -r '.network.executor.ip' config.json)
GPU_ASSIGNER_PORT=$(jq -r '.network.ports.gpu_assigner' config.json)

nc -zv $EXECUTOR_IP $GPU_ASSIGNER_PORT

# From executor node, test connection to controller
CONTROLLER_IP=$(jq -r '.network.controller.ip' config.json)

nc -zv $CONTROLLER_IP $GPU_ASSIGNER_PORT

    Step 4: Start Training Processes

    IMPORTANT: Training processes must be started in a specific order: Controller node first, then Executor node(s).

    4.1 Set Environment Variables

    On all nodes, set the required environment variables:

    # Extract controller information
CONTROLLER_IP=$(jq -r '.network.controller.ip' config.json)
GPU_ASSIGNER_PORT=$(jq -r '.network.ports.gpu_assigner' config.json)

# Set AGA environment variables
export AGA_GPU_ALLOC_SERVER_ADDRESS="$CONTROLLER_IP"
export AGA_GPU_ALLOC_SERVER_PORT="$GPU_ASSIGNER_PORT"

# Verify environment variables are set
echo "AGA_GPU_ALLOC_SERVER_ADDRESS=$AGA_GPU_ALLOC_SERVER_ADDRESS"
echo "AGA_GPU_ALLOC_SERVER_PORT=$AGA_GPU_ALLOC_SERVER_PORT"

    4.2 Start Controller Training Process

    On the controller node (rank 0), start the training process:

    # Extract training configuration
CONTROLLER_IP=$(jq -r '.network.controller.ip' config.json)
MASTER_PORT=$(jq -r '.network.ports.rdzv' config.json)
SCRIPT_PATH=$(jq -r '.training.script_path' config.json)
EPOCHS=$(jq -r '.training.total_epochs' config.json)
SAVE_EVERY=$(jq -r '.training.save_every' config.json)
NNODES=$(jq -r '.training.nnodes' config.json)
NPROC=$(jq -r '.training.nproc_per_node' config.json)
RDZV_ID=$(jq -r '.training.rdzv_id' config.json)

# Set node rank for controller
RANK=0

echo "Starting training on controller node (rank $RANK)..."

# Launch training with AGA
agarun -- torchrun \
    --nproc_per_node="$NPROC" \
    --nnodes="$NNODES" \
    --node_rank="$RANK" \
    --master_addr="$CONTROLLER_IP" \
    --master_port="$MASTER_PORT" \
    --rdzv_id="$RDZV_ID" \
    "$SCRIPT_PATH" \
    "$EPOCHS" \
    "$SAVE_EVERY"

    4.3 Start Executor Training Process

    On the executor node (rank 1), after the controller has started, start the training process:

    # Extract training configuration (same as controller)
CONTROLLER_IP=$(jq -r '.network.controller.ip' config.json)
MASTER_PORT=$(jq -r '.network.ports.rdzv' config.json)
SCRIPT_PATH=$(jq -r '.training.script_path' config.json)
EPOCHS=$(jq -r '.training.total_epochs' config.json)
SAVE_EVERY=$(jq -r '.training.save_every' config.json)
NNODES=$(jq -r '.training.nnodes' config.json)
NPROC=$(jq -r '.training.nproc_per_node' config.json)
RDZV_ID=$(jq -r '.training.rdzv_id' config.json)

# Set node rank for executor
RANK=1

echo "Starting training on executor node (rank $RANK)..."

# Launch training with AGA
agarun -- torchrun \
    --nproc_per_node="$NPROC" \
    --nnodes="$NNODES" \
    --node_rank="$RANK" \
    --master_addr="$CONTROLLER_IP" \
    --master_port="$MASTER_PORT" \
    --rdzv_id="$RDZV_ID" \
    "$SCRIPT_PATH" \
    "$EPOCHS" \
    "$SAVE_EVERY"

    Step 5: Monitoring and Management

    5.1 Monitor Training Progress

    Check training status on any node:

    # Check running training processes
ps aux | grep -E "(torchrun|python.*training)"

# Monitor GPU usage
nvidia-smi

# Check AGA logs
tail -f ~/.acb/gpu-assigner.*.log

# Monitor network connections
ss -tlnp | grep -E "(11234|29400)"

    5.2 Check GPU Assigner Status

    Verify GPU Assigner service status:

    # Check service status
gpu_assigner status

# View detailed process information
ps aux | grep gpu_assigner

# Check log files for errors
tail -f ~/.acb/gpu-assigner.*.log

    Step 6: Shutdown Procedure

    6.1 Stop Training Processes

    Training processes typically stop automatically when training completes. To manually stop:

    # Find training processes
ps aux | grep torchrun

# Stop gracefully (replace PID with actual process ID)
kill -TERM <PID>

# Force stop if necessary (last resort)
kill -9 <PID>

    6.2 Stop GPU Assigner Services

    Stop GPU Assigner services in order: Controller first, then Executor(s):

    On controller node: 

    gpu_assigner --multinode stop

# Verify it stopped
gpu_assigner status

    On executor node: 

    gpu_assigner --multinode stop

# Verify it stopped
gpu_assigner status

    6.3 Clean Up Resources

    Remove any temporary files if needed:

    # Optional: Clean up PID files
rm -f ~/.acb/gpu-assigner.*.pid

# Optional: Archive log files
mv ~/.acb/gpu-assigner.*.log ~/.acb/logs/archive/

    Troubleshooting

    1. Service fails to start with "Port already in use" error

    Likely cause: Port Conflicts

    Solution: 

    # Check what's using the port
sudo lsof -i :11234
sudo lsof -i :29400

# Kill conflicting process (replace <PID> with actual process ID)
sudo kill <PID>

# Or change ports in config.json to use different port numbers

    2. Nodes cannot connect to each other

    Likely cause: Connection Refused Errors

    Solution: 

    # Test network connectivity
ping <target_node_ip>
nc -zv <target_node_ip> <port>

# Check firewall settings
sudo ufw status
sudo iptables -L

# Verify services are listening
ss -tlnp | grep <port>

    3. Services or training fail to start or coordinate

    Likely cause: Wrong Startup Order

    Solution:

    Step 1: GPU Assigner Services

    1. Start executor nodes first
    2. Start controller node second

    Step 2: Training Processes

    1. Start controller node first
    2. Start executor nodes second

    Recovery: Stop all services completely, then restart following steps above

    4. Services fail to start or find each other

    Likely cause: Configuration File Errors

    Solution: 

    # Validate JSON syntax
jq -e '.' config.json

# Check required fields exist
jq -r '.network.controller.ip // "MISSING"' config.json
jq -r '.network.executor.ip // "MISSING"' config.json

# Test IP addresses are reachable
ping $(jq -r '.network.controller.ip' config.json)
ping $(jq -r '.network.executor.ip' config.json)

    5. Training starts but cannot allocate GPUs

    Likely cause: GPU Assignment Issues

    Solution: 

    # Check GPU status
nvidia-smi

# Verify environment variables
echo $AGA_GPU_ALLOC_SERVER_ADDRESS
echo $AGA_GPU_ALLOC_SERVER_PORT

# Check service logs
tail -f ~/.acb/gpu-assigner.*.log

    Log File Locations

    Important log files for debugging:

    • GPU Assigner Logs: ~/.acb/gpu-assigner.<hostname>.log
    • Training Output: Console output from torchrun commands
    • System Logs: /var/log/syslog (for system-level issues)

    Getting Help

    If you encounter issues not covered in this troubleshooting section:

    1. Check the log files for detailed error messages
    2. Verify all prerequisite software is installed correctly
    3. Ensure network connectivity between all nodes
    4. Confirm the correct startup sequence was followed
    5. Validate configuration file syntax and content

    This manual provides a comprehensive guide for setting up multi-node distributed training with ACB/AGA. Follow the steps in order and refer to the troubleshooting section if you encounter any issues.
